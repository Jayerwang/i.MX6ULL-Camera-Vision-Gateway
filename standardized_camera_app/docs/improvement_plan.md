# 规范化和功能设计改进建议

## 当前项目的主要问题

1. 目录职责不清

当前根目录和 `app/` 目录里混放了源码、YUV 原始数据、JPG 转换结果、说明文档和板级 DTS。短期调试很方便，但后续维护会遇到几个问题：不知道哪些文件是源码，哪些是运行产物；采集文件体积大，容易误提交；新同学接手时需要先猜目录含义。

建议改为：

```text
src/        # 源码
include/    # 头文件
docs/       # 文档
scripts/    # 辅助工具
output/     # 运行输出，默认 git 忽略
build/      # 编译输出，默认 git 忽略
```

这样改的原因是把“工程资产”和“运行产物”分开，项目可读性和可维护性会明显提升。

2. 单文件过大，职责耦合

原来的 `ov5640_capture.c` 同时负责命令行解析、V4L2 初始化、缓冲区管理、采集循环、文件写入和错误处理。功能继续增加时，例如增加 JPG 编码、按帧保存、网络推流、AI 推理，就会越来越难改。

建议拆分为：

- `main.c`：只负责参数解析和主流程编排。
- `v4l2_capture.c`：负责 V4L2 设备、缓冲区、采集循环。
- `camera_capture.h`：定义配置结构体、上下文结构体和公共接口。

这样改的原因是让业务逻辑更像流程图，底层硬件操作集中封装，后续扩展时不用在一个大文件里来回穿插。

3. 参数设计和实现不一致

原程序帮助信息里有 `-d DEV`，但实际没有处理 `case 'd'`，设备节点仍然固定为 `/dev/video1`。这会让使用者误以为可以指定 `/dev/video0`，实际运行却不生效。

建议把设备节点放入配置结构体，并真正支持 `-d`。

这样改的原因是命令行参数必须和实际行为一致，否则排查问题会浪费大量时间。

4. 全局变量过多

原程序用多个全局变量保存 fd、缓冲区、宽高、帧率、输出文件等。小程序可以接受，但业务变复杂后，全局状态会让函数依赖关系变得隐蔽。

建议使用：

- `camera_config_t` 保存用户配置。
- `camera_context_t` 保存运行时资源，例如 fd、mmap 缓冲区。

这样改的原因是配置和运行状态边界更清楚，也方便以后做多摄像头或单元测试。

5. 采集产物不应直接放源码目录

`.yuv` 和 `.jpg` 属于运行产物，应该进入 `output/`，并在 `.gitignore` 忽略。

这样改的原因是原始视频帧体积大，且每次运行都会变化，放在源码目录会污染版本管理。

## 功能实现层面的改进方向

1. 第一阶段：稳定采集

- 支持设备节点、分辨率、格式、帧率、帧数、输出路径。
- 检查设备是否支持 `VIDEO_CAPTURE` 和 `STREAMING`。
- 使用 mmap 缓冲区。
- 采集结束后完整释放资源。

2. 第二阶段：输出标准化

- 支持输出单个连续 `.yuv`。
- 支持按帧保存：`frame_001.yuv`、`frame_002.yuv`。
- 提供脚本把 YUYV422 转成 JPG/PNG。
- 在文件名中带上分辨率和格式，例如 `capture_640x480_yuyv422_10f.yuv`。

3. 第三阶段：面向业务扩展

如果这个项目后续用于 AI 网关或边缘视觉，可以继续扩展：

- 采集线程：只负责从摄像头取帧。
- 处理线程：负责格式转换、缩放、AI 推理。
- 输出线程：负责保存、显示或网络发送。
- 帧队列：解耦采集速度和处理速度。

这样设计的原因是摄像头采集、图像处理和业务输出的实时性要求不同，线程和队列可以避免一个环节阻塞整个链路。

## 推荐的业务逻辑流程

```text
解析参数
  -> 打开设备
  -> 查询设备能力
  -> 设置格式和帧率
  -> 申请 mmap 缓冲区
  -> 缓冲区入队
  -> 开启视频流
  -> 循环等待帧
  -> 出队读取帧
  -> 写入文件或交给业务处理
  -> 缓冲区重新入队
  -> 停止视频流
  -> 释放资源
```

这个流程和 V4L2 的生命周期一致，后续定位问题也更容易。
