# OV5640 V4L2 项目学习路线

这份路线用于边搭项目边学习。每个阶段包含：目标、要写的代码、理解检查、作业和复盘点。

## 阶段 0：先跑通原项目

目标：

- 知道这个项目解决什么问题。
- 理解 V4L2 摄像头采集的基本流程。
- 能编译并运行一次采集程序。

你要掌握：

- `/dev/video1` 是 Linux 下的摄像头设备节点。
- `ioctl` 是用户态程序和内核驱动沟通的主要方式。
- `mmap` 是把内核缓冲区映射到用户空间，减少拷贝。
- `.yuv` 是原始图像数据，不是压缩图片。

作业：

1. 用自己的话画出原程序流程图。
2. 说明 `VIDIOC_QUERYCAP`、`VIDIOC_S_FMT`、`VIDIOC_REQBUFS`、`VIDIOC_STREAMON`、`VIDIOC_DQBUF`、`VIDIOC_QBUF` 分别做什么。
3. 解释为什么采集 10 帧的 YUV 文件大小约等于 `width * height * 2 * 10`。

复盘标准：

- 能说清楚 V4L2 的采集生命周期。
- 能说清楚 YUYV422 为什么每像素约 2 字节。

## 阶段 1：整理成标准工程

目标：

- 把单文件实验程序整理成可维护工程。
- 区分配置、运行状态、底层采集和业务入口。

你要写的代码：

- `include/camera_capture.h`
- `src/main.c`
- `src/v4l2_capture.c`
- `Makefile`

重点理解：

- `camera_config_t` 表示用户想要什么。
- `camera_context_t` 表示程序运行时拥有什么资源。
- `main.c` 不直接碰 V4L2 细节，只编排流程。
- `v4l2_capture.c` 封装 Linux/V4L2 细节。

作业：

1. 给 `camera_config_t` 增加一个 `int timeout_sec` 字段。
2. 增加命令行参数 `-t SEC`。
3. 把采集循环里的 `select` 超时时间从固定 2 秒改成用户配置。
4. 修改 README 参数表。

复盘标准：

- 参数从命令行进入 `main.c`，再进入 `camera_context_t`，最后影响 `select`。
- 没有重新引入全局变量。

## 阶段 2：增强输出能力

目标：

- 从“连续写一个 YUV 文件”升级到“可选择输出模式”。

建议功能：

- 模式 1：连续输出到一个 `.yuv` 文件。
- 模式 2：每帧单独保存为 `frame_001.yuv`。

建议设计：

```c
typedef enum {
    OUTPUT_SINGLE_FILE,
    OUTPUT_FRAME_FILES
} output_mode_t;
```

作业：

1. 增加 `output_mode_t`。
2. 增加参数 `-m single|frames`。
3. 实现按帧保存。
4. 如果输出目录不存在，给出清晰错误提示。

复盘标准：

- 输出逻辑不要污染 V4L2 设备配置逻辑。
- 文件命名要可预测。

## 阶段 3：错误处理和日志

目标：

- 让程序更像生产工具，而不是实验脚本。

建议功能：

- 统一错误返回。
- 打印设备实际设置后的宽高、格式和帧大小。
- 增加日志等级：error、warn、info、debug。

作业：

1. 封装 `log_info()`、`log_warn()`、`log_error()`。
2. 增加 `-v` 打开 debug 日志。
3. 所有错误信息都包含操作名和 `strerror(errno)`。

复盘标准：

- 出错时用户知道失败在哪一步。
- 正常运行时日志不吵。

## 阶段 4：业务化设计

目标：

- 从采集 demo 走向边缘视觉应用。

推荐架构：

```text
capture thread -> frame queue -> process thread -> output thread
```

线程职责：

- 采集线程：只负责从 V4L2 拿帧。
- 处理线程：格式转换、缩放、AI 推理。
- 输出线程：保存、显示、推流或上报结果。

作业：

1. 先不用真正开线程，画出数据结构。
2. 设计 `frame_t`，字段至少包含数据指针、大小、宽高、格式、序号、时间戳。
3. 写一个伪代码说明三线程怎么配合。

复盘标准：

- 能说明为什么不把采集、处理和输出全塞进一个循环。
- 能说明队列如何解耦速度不同的模块。

## 理解自测问题

1. 为什么 V4L2 采集要先 `VIDIOC_QBUF` 再 `VIDIOC_STREAMON`？
2. `VIDIOC_DQBUF` 拿到的是什么？
3. 为什么拿完一帧后还要再次 `VIDIOC_QBUF`？
4. `mmap` 比 `read` 方式有什么优势？
5. `camera_config_t` 和 `camera_context_t` 的边界是什么？
6. 为什么输出 JPG 不应该直接混进最底层的 V4L2 初始化代码？
7. 如果摄像头不支持你设置的分辨率，程序应该怎么处理？
8. 如果你想接两个摄像头，现在的结构还缺什么？
