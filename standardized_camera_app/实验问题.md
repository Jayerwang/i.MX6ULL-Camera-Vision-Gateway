问题一：：：mjeg各式采集时bytes=0 sequence=0

不一定代表真的没采到数据，因为当前代码是在 VIDIOC_QBUF 归还 buffer 之后才打印 buf.bytesused，驱动可能已经把这个结构里的字段清掉了。，所以你看到 0，但不一定代表文件没写进去。
改动：
现在会在 VIDIOC_QBUF 之前先保存：
bytesused = buf.bytesused;
sequence = buf.sequence;
并且新增了统计输出：
Elapsed: x.xxx sec, fps: xx.xx, total: xxx bytes, avg frame: xxx bytes



问题二：：：MJPG 采集成功了，但实际采集效率很低，只有 3.49 fps。
可能原因：
1. fwrite 写文件太慢，阻塞了采集循环
2. 板子存储写入慢
3. USB 带宽或 USB 控制器压力大
4. i.MX6ULL 对 720p MJPEG 处理/搬运能力有限
5. 当前程序单线程，采集和写文件绑在一起
6. select timeout 说明某些时刻驱动没有及时返回帧
尤其你现在是：
DQBUF -> fwrite -> QBUF
也就是说，取到一帧后，先写文件。写完才把 buffer 还给驱动。
如果写文件慢，驱动可用 buffer 就变少，采集就会卡。

现在这个结果怎么判断
结论是：
MJPEG 格式支持没问题
720p MJPEG 单线程写文件模式下不流畅
瓶颈很可能在“写文件 + 单线程采集”
不是格式解析失败，也不是 bytes=0 的问题。

下一步怎么验证瓶颈
建议你先跑两个对比测试。
不保存到慢路径，写到 /tmp
./ov5640_capture -d /dev/video1 -w 1280 -h 720 -f MJPG -r 30 -n 100 -o /tmp/mjpg_720p.mjpg
如果 fps 明显升高，说明 /root 所在存储写入慢。
降低分辨率测试
./ov5640_capture -d /dev/video1 -w 640 -h 480 -f MJPG -r 30 -n 100 -o /tmp/mjpg_640x480.mjpg
如果 640x480 接近 30fps，而 1280x720 很低，说明 720p 对当前链路压力太大。
测 YUYV 640x480
./ov5640_capture -d /dev/video1 -w 640 -h 480 -f YUYV -r 30 -n 100 -o /tmp/yuyv_640x480.yuv

1280x720 MJPG 保存到 /root：1.77 到 3.49 fps，很慢
640x480 MJPG 保存到 /tmp：20.06 fps，明显正常很多
640x480 YUYV：连续 timeout，当前不适合作为主线

结论比较清楚：
你的 USB 摄像头 MJPG 是可用的。
640x480 MJPG 比较稳。
1280x720 MJPG 在当前板子/USB/写文件链路下压力较大。
YUYV 路径当前异常或非常不稳定，先不要作为主线。
sequence 跳得很大，比如到 1018，说明驱动/摄像头那边产生过很多帧，但你的程序实际只拿到 100 帧，中间有大量帧没被应用层及时消费。这个现象通常和这几类有关：
写文件阻塞
USB/驱动缓冲不足
720p 数据压力大
单线程 DQBUF -> fwrite -> QBUF 太慢
